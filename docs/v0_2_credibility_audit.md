v0.2 benchmark line. The goal is not to add features; it is to make every public
benchmark claim reproducible, challengeable, and evidence-labeled.

## What Is Already Strong Enough

- The v0.2 comparison contract is frozen and hashable.
- The PyPI and GitHub Release distribution hashes for `0.2.0` match the tracked
  release artifact inventory.
- Fresh install behavior is now part of CI rather than a maintainer-only local
  habit.
- The benchmark has non-LLM baselines and evidence labels, so model rows are not
  presented as naked profitability rankings.
- The execution simulator is explicitly labeled as `stress-only` unless a row
  attaches quote/fill provenance.

## What Is Not Yet Strong Enough

- External reproduction is still pending. Maintainer CI does not replace three
  independent environment reports.
- The public quote/fill sample is useful calibration plumbing, but it is not
  broad enough to support venue-wide or broker-grade transaction-cost claims.
- Provider-backed model rows are mostly `cached-provider` and `redacted-prompt`.
  They should support reliability/autopsy claims, not broad model-skill claims.
- Exact immutable release parity now requires either leaving `v0.2.0` as the
  published protocol release and treating the benchmark-card cleanup as
  post-release hardening, or publishing a patch release after this cleanup.

## Next Maintainer Actions

1. Close the release-alignment loop with a patch release if exact tag/PyPI/docs
   parity is required for citations.
2. Recruit or run independent fresh-environment reports for #43, #44, and #45.
3. Prioritize #47 over new model rows: a small external quote/fill calibration
   report is more valuable than another cached provider comparison.
4. Review #48 before making stronger scientific claims in README, papers, or
   leaderboard captions.
5. Treat the benchmark card as an evidence table: claim badges first,
   performance ranking second.
